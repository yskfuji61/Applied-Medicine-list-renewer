Signing / Notarization / Provenance

署名・公証・Provenance 実装手順書

この文書は、薬剤リスト変換アプリの個人配布運用を「未署名 ZIP の配布」から「署名済みバイナリ + 公証済み macOS 配布物 + 検証可能な checksum / provenance asset を持つ release」へ引き上げるための実装レベルの手順書です。単なる概念説明ではなく、どの script と workflow に何を入れ、どの secret 名を使い、どの順番で build / sign / verify / upload するかまでを固定します。

完了条件

macOSDeveloper ID Application で署名され、notarytool で notarization が完了し、staple 済みの .app から ZIP を再生成する。
WindowsPyInstaller が出力した .exe と同梱 PE ファイルを Authenticode 署名し、Get-AuthenticodeSignature で有効署名を確認してから ZIP 化する。
Release IntegritySHA256SUMS.txt とその署名ファイル、または provenance artifact を release asset として添付する。release notes の本文だけに checksum を置かない。
WorkflowCI から生成された配布物は、同じ job 内で署名・検証・checksum 生成・asset upload まで完了する。
運用配布者以外が、OS 標準検証または公開鍵で改ざん有無を検証できる。
この手順書の狙いは「見た目の release 品質」ではなく、「受け手が正当な配布者と改ざん有無を検証できること」です。

実装方針

macOS

Developer ID + notarization を build pipeline に統合する

現行 repo には scripts/sign_and_notarize_macos.sh と .github/workflows/macos-standalone-release.yml が既にあります。これを「build 後の任意手順」から「release build の正規工程」へ格上げし、ZIP の再生成方法も metadata 汚染しない形へ修正します。

Windows

AuthentiCode は build script 内で完結させる

現行の build_windows_standalone.ps1 は build と zip 化だけです。ここに署名処理と署名検証を直接組み込みます。推奨は CI では Azure Trusted Signing、ローカル release machine では PFX 証明書です。

Integrity

release notes ではなく asset を増やす

最低限追加する asset は SHA256SUMS.txt と、その署名ファイルです。ファイル名は ASCII を使います。GitHub Releases は日本語 asset 名を内部保存名で正規化するため、integrity 用の asset まで Unicode にしない方が運用事故が少ないです。

重要

署名は最後に行う

アイコン差し替え、README コピー、config 書き換え、ZIP 再生成のあとに署名するのではありません。ファイル内容が全て確定したあとで app / exe / dll を署名し、その後は ZIP 包装と checksum 生成だけに限定します。

Secret / 証明書インベントリ

macOS localAPPLE_CERT_P12_PATH, APPLE_CERT_P12_PASSWORD, APPLE_SIGN_IDENTITY, APPLE_ID, APPLE_TEAM_ID, APPLE_APP_SPECIFIC_PASSWORD, APPLE_NOTARY_PROFILE
macOS ActionsMACOS_CERTIFICATE_BASE64, MACOS_CERTIFICATE_PASSWORD, KEYCHAIN_PASSWORD, APPLE_SIGN_IDENTITY, APPLE_ID, APPLE_APP_SPECIFIC_PASSWORD, APPLE_TEAM_ID
Windows localWINDOWS_CERT_PFX_PATH, WINDOWS_CERT_PFX_PASSWORD, WINDOWS_SIGN_SUBJECT, WINDOWS_TIMESTAMP_URL
Windows Actions 推奨Azure Trusted Signing 用の tenant / client / account / profile 情報、または CI 用 PFX を Base64 化した WINDOWS_CERTIFICATE_BASE64 と WINDOWS_CERTIFICATE_PASSWORD
checksum signingMINISIGN_SECRET_KEY または GPG 秘密鍵。個人配布なら offline 検証が簡単な minisign を推奨。
現行の GitHub Release は notes に SHA256 を記載していますが、それだけでは「改ざんされていない checksum 自体」を証明できません。checksum そのものに署名が必要です。

Phase 1: macOS Developer ID 署名と notarization を build に組み込む

1

現状の差分を理解する

現行の scripts/build_macos_standalone.sh は build 後に codesign --sign - を打っていますが、これは ad-hoc 署名であり配布者証明には使えません。さらに scripts/sign_and_notarize_macos.sh は ZIP 再生成に ditto --sequesterRsrc を使っており、以前除去した macOS metadata を再混入させるリスクがあります。

この 2 点はそのまま運用に入れてはいけません。macOS 側は「Developer ID 署名に一本化」「notarization 後の ZIP 再生成は metadata 汚染しない方法に変更」が必須です。
2

build script の責務を整理する

次の 3 file を対象にします。

  1. scripts/build_macos_standalone.sh
  2. scripts/build_macos_app.sh
  3. scripts/sign_and_notarize_macos.sh

責務分離は次で固定します。

build scriptunsigned だが内容確定済みの release directory と ZIP を作る。署名はしない。
sign scriptapp を署名し、検証し、ZIP を再生成し、notarization を通し、staple し、検証し、checksum を生成する。
workflowbuild script と sign script を順に呼び、署名済み ZIP と checksum asset を upload する。
3

build_macos_standalone.sh / build_macos_app.sh の改修要件

  1. ad-hoc 署名行を削除する。
  2. ZIP 生成は現行どおり zip -X -r -y を維持する。
  3. 必要なら SIGN_AND_NOTARIZE=1 時だけ sign script を呼べるようにするが、責務は sign script 側に置く。
  4. wrapper を今後も配るなら、wrapper と standalone の両方で同じ署名フローを使えるように APP_DIR / RELEASE_DIR / ZIP_PATH を外から受けられる構造にする。
export SIGN_AND_NOTARIZE=1
export MACOS_RELEASE_KIND=standalone
./scripts/build_macos_standalone.sh

# build script の最後で呼ぶ場合のイメージ
if [[ "${SIGN_AND_NOTARIZE:-0}" == "1" ]]; then
  APP_DIR="$APP_DIR" \
  RELEASE_DIR="$RELEASE_DIR" \
  ZIP_PATH="$ZIP_PATH" \
  ./scripts/sign_and_notarize_macos.sh
fi
4

sign_and_notarize_macos.sh の改修要件

現行 script は土台として使えますが、次を必須変更とします。

  1. APP_DIR, RELEASE_DIR, ZIP_PATH, SHA_PATH を環境変数で上書き可能にする。
  2. ditto --sequesterRsrc を廃止し、staple 完了後に zip -X -r -y で release directory から再包装する。
  3. codesign --verify に加えて spctl -a -vvv を notarization 後にも必ず実行する。
  4. ZIP と checksum のみを asset 化し、release notes 本文への依存をなくす。
#!/bin/zsh
set -euo pipefail

APP_DIR=${APP_DIR:-"$DIST_DIR/$RELEASE_NAME/薬剤リスト変換アプリ.app"}
RELEASE_DIR=${RELEASE_DIR:-"$DIST_DIR/$RELEASE_NAME"}
ZIP_PATH=${ZIP_PATH:-"$DIST_DIR/薬剤リスト変換アプリ-standalone-macos.zip"}
SHA_PATH=${SHA_PATH:-"$DIST_DIR/SHA256SUMS.txt"}

xattr -cr "$APP_DIR"
codesign --force --deep --options runtime --timestamp --sign "$APPLE_SIGN_IDENTITY" "$APP_DIR"
codesign --verify --deep --strict --verbose=2 "$APP_DIR"

TMP_NOTARY_ZIP="$DIST_DIR/notary-upload.zip"
rm -f "$TMP_NOTARY_ZIP"
(cd "$DIST_DIR" && zip -X -r -y "$TMP_NOTARY_ZIP" "$(basename "$RELEASE_DIR")" >/dev/null)

xcrun notarytool submit "$TMP_NOTARY_ZIP" --keychain-profile "$APPLE_NOTARY_PROFILE" --wait
xcrun stapler staple "$APP_DIR"
spctl -a -vvv "$APP_DIR"

rm -f "$ZIP_PATH"
(cd "$DIST_DIR" && zip -X -r -y "$ZIP_PATH" "$(basename "$RELEASE_DIR")" >/dev/null)
notarization 用の一時 ZIP と最終配布 ZIP は分ける方が安全です。submit に使った ZIP をそのまま release asset に固定すると、後段で checksum を作り直したくなったときに運用がぶれます。
5

macOS workflow の変更点

  1. .github/workflows/macos-standalone-release.yml の notarize path を既定動線にする。
  2. 署名済み ZIP と SHA256SUMS.txt, SHA256SUMS.txt.minisig を同時に upload する。
  3. 必要なら wrapper 用 job を別追加するが、最初は standalone のみを正規 release とする。
- name: Build standalone release

Phase 2: Windows Authenticode 署名を build に組み込む

6

Windows は build_windows_standalone.ps1 内に署名を入れる

署名を workflow だけに置くと、ローカルで同じ配布物を再現できません。scripts/build_windows_standalone.ps1 に署名ブロックを入れ、workflow からもローカル release machine からも同じ script を使う形にします。

署名対象.exe だけでなく、bundle 配下の .dll と .pyd も対象にする。
タイミングPyInstaller build 後、ZIP 化前。
検証Get-AuthenticodeSignature で Status -eq Valid を必須にする。
$SignEnabled = $env:WINDOWS_SIGN_ENABLED -eq '1'
$TimestampUrl = $env:WINDOWS_TIMESTAMP_URL

if ($SignEnabled) {
  $targets = Get-ChildItem -Path $FinalAppDir -Recurse -File |
    Where-Object { $_.Extension -in '.exe', '.dll', '.pyd' }

  foreach ($target in $targets) {
    & signtool sign /fd SHA256 /td SHA256 /tr $TimestampUrl \
      /f $env:WINDOWS_CERT_PFX_PATH /p $env:WINDOWS_CERT_PFX_PASSWORD \
      $target.FullName

    $signature = Get-AuthenticodeSignature -FilePath $target.FullName
    if ($signature.Status -ne 'Valid') {
      throw "Authenticode verification failed: $($target.FullName) status=$($signature.Status)"
    }
  }
}
7

CI は Azure Trusted Signing を第一候補にする

PFX を CI に持ち込むより、GitHub Actions で Azure Trusted Signing を使う方が、秘密鍵の保護と失効運用の面でましです。個人配布でも Windows の SmartScreen 対策を継続するならこちらを推奨します。

推奨GitHub Actions では Trusted Signing action または signtool + Trusted Signing endpoint を使う。
fallbackローカル release machine では PFX 証明書を使う。
避ける未署名の Windows standalone ZIP を release に載せること。
- name: Sign Windows bundle
  uses: azure/trusted-signing-action@v0
  with:
    azure-tenant-id: ${{ secrets.AZURE_TENANT_ID }}
    azure-client-id: ${{ secrets.AZURE_CLIENT_ID }}
    azure-client-secret: ${{ secrets.AZURE_CLIENT_SECRET }}
    endpoint: ${{ secrets.AZURE_TRUSTED_SIGNING_ENDPOINT }}
    trusted-signing-account-name: ${{ secrets.AZURE_TRUSTED_SIGNING_ACCOUNT }}
    certificate-profile-name: ${{ secrets.AZURE_TRUSTED_SIGNING_PROFILE }}
    files-folder: dist/windows-standalone-release/薬剤リスト変換アプリ
PFX 方式を選ぶ場合でも、最終的な release asset 生成を CI に寄せるなら、証明書は repository secret ではなく organization secret か environment secret に置く方がよいです。

Windows workflow の変更点

  1. .github/workflows/windows-standalone-release.yml の build step の前後に sign / verify を入れる。
  2. 署名後の ZIP に対して現行の forbidden entry 監査を維持する。
  3. 最後に checksum / signature asset も upload する。
- name: Build standalone release
  shell: pwsh
  env:
    WINDOWS_SIGN_ENABLED: '1'
    WINDOWS_TIMESTAMP_URL: ${{ secrets.WINDOWS_TIMESTAMP_URL }}
    WINDOWS_CERT_PFX_PATH: ${{ runner.temp }}\release-signing.pfx
    WINDOWS_CERT_PFX_PASSWORD: ${{ secrets.WINDOWS_CERTIFICATE_PASSWORD }}
  run: .\scripts\build_windows_standalone.ps1

- name: Verify Authenticode status
  shell: pwsh
  run: |
    Get-ChildItem dist/windows-standalone-release/薬剤リスト変換アプリ -Recurse -File |
      Where-Object { $_.Extension -in '.exe', '.dll', '.pyd' } |
      ForEach-Object {
        $sig = Get-AuthenticodeSignature $_.FullName
        if ($sig.Status -ne 'Valid') { throw "Invalid signature: $($_.FullName)" }
      }

Phase 3: signed checksum file または provenance asset を release に添付する

9

推奨構成

必須 assetSHA256SUMS.txt, SHA256SUMS.txt.minisig
推奨 assetrelease-manifest.json。各配布物の SHA256、サイズ、build run id、commit SHA、OS、署名方式を入れる。
任意追加release-provenance.intoto.jsonl または GitHub Artifact Attestation。CI build provenance を強くしたい場合に追加。
命名checksum / provenance asset は ASCII 名に統一する。GitHub Releases の Unicode asset name 正規化問題を避けるため。
10

checksum 生成と署名の標準手順

個人配布では、受け手が追加ツールなしで ZIP を取得し、必要なら軽量に verify できる構成がよいです。ここでは minisign を標準とします。

#!/bin/zsh
set -euo pipefail

DIST_DIR=${1:-dist}
CHECKSUM_FILE="$DIST_DIR/SHA256SUMS.txt"

shasum -a 256 \
  "$DIST_DIR/薬剤リスト変換アプリ-macos.zip" \
  "$DIST_DIR/薬剤リスト変換アプリ-standalone-macos.zip" \
  "$DIST_DIR/薬剤リスト変換アプリ-standalone-windows.zip" \
  > "$CHECKSUM_FILE"

minisign -S \
  -s "$MINISIGN_SECRET_KEY" \
  -m "$CHECKSUM_FILE" \
  -x "$DIST_DIR/SHA256SUMS.txt.minisig" \
  -t "Applied-Medicine-list-renewer release checksums"

受け手向けの検証手順は次です。

minisign -Vm SHA256SUMS.txt -P '公開鍵文字列'
shasum -a 256 -c SHA256SUMS.txt

Release notes は説明に使い、検証用データは asset に寄せること。notes だけで integrity を成立させないこと。

Phase 4: 検証ゲート

Gate 1

Build 完了

3 配布物が生成され、実データ非同梱、不要 metadata 非同梱であること。

Gate 2

OS 署名検証

macOS は codesign --verify と spctl、Windows は Get-AuthenticodeSignature が通ること。

Gate 3

Reconstruction

ZIP 展開後に .app / .exe が再構成されること。既存の Windows audit step は継続利用する。

Gate 4

Integrity Assets

SHA256SUMS.txt と署名ファイルが release asset として添付され、検証コマンドが通ること。

# macOS
codesign --verify --deep --strict --verbose=2 'dist/macos-standalone-release/薬剤リスト変換アプリ.app'
spctl -a -vvv 'dist/macos-standalone-release/薬剤リスト変換アプリ.app'

# Windows
Get-AuthenticodeSignature 'dist\windows-standalone-release\薬剤リスト変換アプリ\薬剤リスト変換アプリ.exe'

# checksum
minisign -Vm dist/SHA256SUMS.txt -P '公開鍵文字列'
shasum -a 256 -c dist/SHA256SUMS.txt

運用順序の固定版

  1. テストを通す。
  2. unsigned build を生成する。
  3. macOS app を Developer ID 署名し、公証し、staple する。
  4. Windows bundle を Authenticode 署名し、全署名対象を検証する。
  5. ZIP を再生成する。
  6. SHA256SUMS.txt と署名ファイルを生成する。
  7. ZIP 展開と再構成監査を行う。
  8. Release asset を upload する。
  9. Release page で asset 表示名と checksum asset の存在を目視確認する。
この順序を崩すと、署名済みのはずの app を後から書き換えてしまう、checksum が未署名 artifact を指す、notarization 後に ZIP 内容が変わる、といった事故が起きます。

この repo に対する具体的な変更対象

改修scripts/build_macos_standalone.sh から ad-hoc 署名を削除し、必要なら sign script 呼び出しだけを残す。
改修scripts/build_macos_app.sh にも同じ責務分離を入れる。wrapper を配布しないなら release 対象から外す。
改修scripts/sign_and_notarize_macos.sh の ZIP 再生成を metadata 汚染しない形へ変更する。
改修scripts/build_windows_standalone.ps1 に Authenticode 署名と検証を追加する。
追加scripts/generate_signed_checksums.sh または同等 script を追加する。
改修.github/workflows/macos-standalone-release.yml と .github/workflows/windows-standalone-release.yml に checksum asset upload を追加する。
追加必要なら .github/workflows/publish-release.yml を新設し、asset の publish を 1 workflow に集約する。

非推奨パターン

  • 未署名 app / exe を GitHub Releases に直接置く。
  • release notes の本文だけに SHA256 を書く。
  • macOS notarization 後に app 内容を書き換える。
  • Windows で outer exe だけ署名して DLL / PYD を未署名のまま残す。
  • Unicode 名の integrity asset を使い、GitHub の内部正規化挙動に依存する。
  • ローカル build と CI build で署名手順が別物になる。

最終判断

この repo は、実データ非同梱と release 監査の面ではかなり整いましたが、配布者証明はまだ未完成です。実装優先順位は明確で、最初に macOS の Developer ID + notarization を build pipeline に統合し、次に Windows Authenticode を同じ粒度で build script に入れ、最後に SHA256SUMS.txt と署名ファイルを release asset として必須化します。ここまで入れば、個人配布でも「誰が配ったか」「改ざんされていないか」を受け手が検証できる水準に到達します。
