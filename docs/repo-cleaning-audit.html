総合判定

判定 公開配布向けではなく、個人利用向けでもまだ「クリーンな配布母体」ではありません。理由は、コード・入力実データ・旧システム由来の参照 CSV・監査生成物・リリース運用スクリプトが同じ履歴に混在しているためです。
最優先 責務分離より先に、実データと生成物を現行履歴から切り離すことです。これを後回しにすると、private 運用でも repo の役割が曖昧になり、将来の共有や再利用時に毎回クリーニングが必要になります。
推奨ゴール この repo は「変換エンジンと最小限の運用ドキュメント」に限定し、入力データ・旧比較用マスタ・生成済みレポート・配布物は外部ワークディレクトリへ退避する構成にします。

主要 Findings

High

実データがコードと同じ履歴に入っている

260508_Musashino_採用医薬品 配下の 3 CSV と 旧採用医薬品リスト 配下の 5 CSV が tracked です。個人配布へ切り替えるなら、配布対象のコード repo に残す理由は弱く、将来の accidental share の温床になります。

High

監査生成物 96 ファイルが tracked

docs/audit-reports に比較履歴、生成 CSV、HTML/JSON レポートが蓄積されています。これは再生成可能な成果物であり、リポジトリの本質的な資産ではありません。

High

構成ファイルが実データ前提に固定されている

config/defaults.json が 旧採用医薬品リスト 内の具体ファイル名へ直接依存しています。コード repo からデータを外す前提なら、設定の抽象化とパス分離が必要です。

Medium

個人配布には過剰な release/signing 面が repo 主体になっている

.github/workflows/macos-standalone-release.yml と各種 signing/notary スクリプトは有効ですが、個人配布では「常設 CI で notarize する」責務を repo 本体から分離したほうが運用事故が減ります。

Medium

README と配布 docs が公開配布寄り

現行の README は配布・署名・GitHub Actions の説明が前面です。個人配布向けに整理するなら、開発 repo 用 README と配布 bundle 用 README の責務を分けるべきです。

Keep

コア変換ロジックは十分に分離済み

src/pharmalist は 12 ファイルでまとまっており、変換エンジン本体として独立可能です。クリーンアップ対象は主に周辺資産です。

残す src/, tests/, scripts/ のうち開発に必要なもの, config/, 最小限の docs/, assets/, pyproject.toml
外へ出す 新入力 CSV、旧採用医薬品リスト、Excel 原本、生成済み監査レポート、配布 ZIP / .app

境界の確定

repo に残す .github/workflows/macos-standalone-release.yml, .gitignore, LICENSE, README.md, assets/, packaging/, pyproject.toml, src/, tests/, docs/requirements-spec.html, docs/macos-distribution-guide.html, この監査 HTML 2 本
残すが先に修正 config/defaults.json, .vscode/tasks.json, launch_pharmalist.command, scripts/build_macos_app.sh, scripts/build_macos_standalone.sh, scripts/sign_and_notarize_macos.sh, scripts/setup_local_release_machine.sh, scripts/configure_github_actions_secrets.sh。いずれも repo 内実データ前提を外してから keep する。
repo 外へ出す 260508_Musashino_採用医薬品/, 旧採用医薬品リスト/, 簡易 医薬品集.xlsx, docs/audit-reports/, dist/
repo 内で場所変更 RELEASE_NOTES_TEMPLATE.md はルート直下ではなく docs/templates/ などのテンプレート領域へ移すのが妥当
この境界により、個人配布向け cleaning の第 1 段階では「実データ・生成物を外に出す」「実データに固定された scripts と config を抽象化する」までを必須範囲とし、workflow や packaging は即削除対象にしません。

priority 順の cleaning 対象

  1. tracked 実データ: 260508_Musashino_採用医薬品, 旧採用医薬品リスト, 簡易 医薬品集.xlsx
  2. tracked 生成物: docs/audit-reports
  3. 設定の実データ固定参照: config/defaults.json
  4. 個人配布に過剰な常設リリース運用: workflow と signing scripts の置き場所・実行条件
  5. README / docs の責務分離
現在の一番大きいリスクは「コードの質」ではなく「資産の混在」です。先にフォルダをきれいに見せるのではなく、履歴と役割を切り分ける設計が必要です。

結論

この repo は、少数ファイルの責務分離だけで clean にはなりません。必要なのは「コード repo」「個人用データ置き場」「配布成果物置き場」を分離し、さらに既存 Git 履歴から実データと監査生成物を消す段階的作業です。

実装手順は別紙の repo-cleaning-implementation-guide.html にまとめます。そちらは、実際のブランチ運用、移設順序、履歴 rewrite、完了判定まで含めた手順書です。
