Implementation Playbook

個人配布向け repo cleaning 実装手順書

この手順書は、現行 repo を壊さずに監査結果を反映するための実装レベルの playbook です。目的は、コード repo を最小化し、個人利用データと生成物を分離し、将来 accidental share しても被害が出にくい構造にすることです。

完了条件

コード repo実データ、旧マスタ、生成済みレポート、配布物が tracked されていない
設定データパスが repo 外ワークスペースを指せる
README開発 repo 用と配布用が分かれている
履歴必要なら実データと監査生成物が Git 履歴から除去されている
検証テスト、最小変換、個人配布パッケージ生成が再現可能

確定した境界

keep .github/, .gitignore, .vscode/tasks.json の枠組み, LICENSE, README.md, assets/, packaging/, pyproject.toml, src/, tests/, docs/requirements-spec.html, docs/macos-distribution-guide.html, cleaning 関連 HTML
keep after refactor config/defaults.json, launch_pharmalist.command, scripts/build_macos_app.sh, scripts/build_macos_standalone.sh, scripts/sign_and_notarize_macos.sh, scripts/setup_local_release_machine.sh, scripts/configure_github_actions_secrets.sh。repo 内の具体データパス依存を除去したあとで keep する。
move out 260508_Musashino_採用医薬品/, 旧採用医薬品リスト/, 簡易 医薬品集.xlsx, docs/audit-reports/, dist/
relocate in repo RELEASE_NOTES_TEMPLATE.md はルートではなく docs/templates/ などへ移動する。配布用 README も将来的に同じ領域へテンプレート化する。
この分類を以後の cleaning 実装の正式な境界とする。迷った場合は「実データに依存するか」「再生成可能か」「コード repo を見せたときに説明コストが高いか」で判定する。

Phase 0: 凍結とバックアップ

0

現行状態を保全する

  1. 新しい cleanup 用ブランチを切る。
  2. 現在の main をタグで退避する。
  3. 実データと配布物を repo 外へフルコピーする。
git checkout -b cleanup/personal-distribution
git tag repo-before-cleaning-20260513
mkdir -p ../pharmalist-private-workspace/{inputs,references,reports,releases,archive}
cp -R 260508_Musashino_採用医薬品 ../pharmalist-private-workspace/inputs/current
cp -R 旧採用医薬品リスト ../pharmalist-private-workspace/references/legacy
cp -R docs/audit-reports ../pharmalist-private-workspace/reports/audit-reports
cp -R dist ../pharmalist-private-workspace/releases/current
履歴 rewrite の前に、repo 外バックアップがない状態で削除を始めないこと。ここを飛ばすと復旧コストが一番高いです。

Phase 1: tracked 資産の切り離し

1

repo 内から remove する対象

  1. 260508_Musashino_採用医薬品
  2. 旧採用医薬品リスト
  3. 簡易 医薬品集.xlsx
  4. docs/audit-reports
git rm -r --cached 260508_Musashino_採用医薬品 旧採用医薬品リスト docs/audit-reports
git rm --cached '簡易 医薬品集.xlsx'

ここでは --cached を使い、まず Git 管理から外します。ファイル実体はワークツリーに残るため、repo 外移設が終わるまで安全です。

2

.gitignore を強化する

# local data workspaces
/260508_Musashino_採用医薬品/
/旧採用医薬品リスト/
/簡易 医薬品集.xlsx
/docs/audit-reports/
/dist/
/docs/templates/release-assets/
/tmp/
/reports/
/releases/
目的は「今後うっかり add されないこと」です。履歴掃除はこの後です。

Phase 2: 設定と責務の分離

3

config/defaults.json をテンプレート化する

現状の config/defaults.json は具体ファイル名へ直結しています。これを repo 内テンプレートとローカル override に分けます。

repo に残すconfig/defaults.template.json またはサンプル設定
local に置く../pharmalist-private-workspace/config/local.json
アプリ側環境変数または CLI オプションで local config を受け取る
export PHARMALIST_CONFIG=../pharmalist-private-workspace/config/local.json
PYTHONPATH=src python3 -m pharmalist.cli publish-report \
  ../pharmalist-private-workspace/inputs/current/references \
  ../pharmalist-private-workspace/references/legacy \
  --name local-run
4

docs の責務を分ける

  1. repo の docs には要件、設計、運用原則だけを残す。
  2. 生成レポートは local workspace の reports/ に出力する。
  3. 配布 bundle 用 README は build 時に生成または別テンプレートから複製する。

Phase 3: packaging / signing 運用の整理

5

個人配布前提に release 運用を縮退する

個人配布では、常設の GitHub Actions workflow は必須ではありません。次の 2 方式のどちらかに寄せます。

方式 Aworkflow は残すが手動専用のまま維持し、README からは隠す
方式 Bworkflow を ops/ 相当へ退避、日常 repo からは外す

個人配布なら方式 A で十分です。scripts 自体は残し、日常の開発経路から見えにくくするだけで事故率は下がります。

Phase 4: 履歴の掃除

6

現行 Git 履歴から除去する

private repo であっても、実データと監査成果物を future-safe にしたいなら履歴 rewrite を推奨します。最も扱いやすいのは git filter-repo です。

brew install git-filter-repo
git filter-repo \
  --path 260508_Musashino_採用医薬品 \
  --path 旧採用医薬品リスト \
  --path '簡易 医薬品集.xlsx' \
  --path docs/audit-reports \
  --invert-paths

rewrite 後は remote を強制更新する必要があります。

git push origin --force --all
git push origin --force --tags
すでに clone 済みの他環境があるなら、rewrite 後は全 clone を作り直す前提で行います。

Phase 5: 検証

7

検証項目

  1. git ls-files に実データと監査生成物が残っていない。
  2. git log --stat で rewrite 対象が見えない。
  3. PYTHONPATH=src python3 -m unittest tests.test_profile が通る。
  4. repo 外 workspace を指して変換が動く。
  5. 必要なら ./scripts/build_macos_standalone.sh が repo 外データ前提でも成立する。
git ls-files | grep -E '260508_Musashino|旧採用医薬品リスト|audit-reports|簡易 医薬品集.xlsx'
PYTHONPATH=src python3 -m unittest tests.test_profile
